**「GPU機と同じ環境を、グラボなしのCPU機で構築・起動できるか」** を、推測でなく実機で判定する。

## ■ 判定

### 結論：**グラボなしCPU機での環境構築・起動は「可能」。GPUの有無は障害ではない。**

- ComfyUI本体はCPU機で完全に起動する（595ノード）。生成だけが遅い（が、それはやらない方針）。
- 出ている失敗は**すべてハードウェア非依存の環境問題**（pip未導入＋カスタムノード同士の名前衝突）。
  - → これらは **GPU機でも同じく壊れる**。直せば両機で直る。
- よって「準備はCPU機、生成はGPU機」という当初目的は、技術的に成立する。

### ただし注意点（当初企画の前提とのズレ）

1. **環境は実は2系統**。当初企画は「venv共通（GPU・CPU共用）」を想定していたが、実機は
   - CPU機（このSSD）= `D:\ai\ComfyUI\venv`
   - GPU機（出先）= `ComfyUI_GPU_NVIDIA.bat` が参照する `python_embeded`（このSSDには存在しない）
   → 「同一環境」は**“同じSSD上の1つのvenv”ではなく、“2つの並行環境を同期維持”** が実態。pip依存はそれぞれに入れる必要がある。
2. **SSDポータブルなので「コードとモデルは移動する」**。ワークフローJSON・カスタムノードのコード・モデルファイルはSSDと一緒にGPU機へ行く。機械ごとに違うのは **pipパッケージ環境とGPUドライバだけ**。
3. `utils`衝突は load 順依存で**現状フル環境は両機で不安定**。フル環境を正とするなら、この衝突解消（該当ノードの隔離/修正/load順制御）が前提作業になる。

## ■ ここから取りうる2つの道

| | 道A：フル環境を直して維持 | 道B：使うワークフローにスコープを絞る |
|---|---|---|
| 方針 | 595ノード全部が通る「正しい共通環境」を作り、CPU機・GPU機の両方で同期維持 | 実際に使うworkflowが要求するノード/モデル/pipだけを抽出し、GPU機はそれだけ満たす |
| CPU機の作業 | utils衝突の解消＋`pandas`等の不足導入＋以後の整合維持 | ワークフローを解析して必要物だけ列挙（グラボ不要） |
| メリット | どのworkflowでも動く。汎用 | 軽い・壊れにくい・GPU機の構築が最小。greenbackは**カスタムノード0＝追加install不要**が判明済み |
| デメリット | 衝突解消と二重管理コスト。使わないノードの不具合にも振り回される | workflowごとに定義が要る（ただし自動生成可） |
| 実装可否 | 可（衝突解消は調査要） | 可（CPU機で実装可能。ノード→提供フォルダ対応付けのみ要作り込み） |

### 補足：道Bの裏付け（実測）

greenbackワークフロー（`0304_01_greenback_*`）の必要物：
- 使用ノード9種すべて**コア**（`CheckpointLoaderSimple/LoraLoader/LoadImage/ImageScale/VAEEncode/CLIPTextEncode×2/KSampler/VAEDecode/SaveImage`）→ **カスタムノード不要**
- モデル2個：`novaOrangeXL_v140.safetensors`（XL）/`granblue-klein9b.safetensors`（LoRA）→ SSD上に実在
- → このworkflowに関しては、上記7件の失敗は**完全に無関係**。GPU機はコアComfyUI＋2モデルで生成可能。
