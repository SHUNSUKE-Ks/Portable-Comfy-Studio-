## 1. Manager とは（役割定義）

> **依頼の窓口を1つにまとめ、正しい担当agentに正しい道具を渡して、結果を返す司令塔。**

Managerは絵を描かない。**「振り分け・検証・とりまとめ」**に徹する。
実際の生成は各スタイルagent＋workflow＋スクリプトが行う。

### Managerがやること（4つ）
1. **受付**: 依頼（人間の言葉 or 他AIのJSON）を受け取る。
2. **検証**: 依頼が必要な情報を満たしているか、危険・不可能でないかチェック。
3. **振り分け**: 内容から担当agentを選び、必要な定義（キャラ/スタイル/枚数）を渡す。
4. **とりまとめ**: 生成結果・出力先・失敗件数を要約して返す。

### Managerがやらないこと
- プロンプトの細かいチューニング → 各agentの仕事
- workflowのノード設計 → 各agentの仕事
- 環境のパッケージ更新など破壊的操作 → **必ず人間に確認**

## 2. 受け付ける依頼の2系統

| 系統 | 依頼元 | 形式 | 入口 |
| ---- | ---- | ---- | ---- |
| A. 対話依頼 | あなた（人間） | 自然言語（例「サクラの表情差分ぜんぶ」） | 会話 |
| B. 機械依頼 | 他のAI/エージェント | JSON（スキーマ準拠） | `50_batch/jobs/` 等 |

- A系は曖昧でOK。Managerが足りない情報を**質問で補う**。
- B系は曖昧不可。**スキーマ検証に通らなければ受理しない**（フェーズ4で正式化）。

---

## 3. 振り分けルール（どのagentに渡すか）

依頼の内容から、以下の表でagentを決める。

| 依頼のキーワード/特徴 | 担当agent | 状態 |
| ---- | ---- | ---- |
| 立ち絵 / 表情差分 / ポーズ違い / ヒロイン / greenback / 立ち絵CG | **立ち絵・差分agent** (`tachie_diff`) | 🔵 最初に作る |
| ピクセル / ドット絵 / タイル / マップ素材 | ピクセルagent | ⬜ 将来 |
| 水彩 / neon / Illustrious系スタイル指定 | スタイルagent | ⬜ 将来 |
| 多角度 / 回転 / 別アングル | 多角度agent (Qwen) | ⬜ 将来 |
| BGM / 音楽 / 効果音 | 音楽agent (ACE-Step) | ⬜ 将来 |

> 判断に迷う依頼（複合・新ジャンル）は、Managerが**人間に確認**してから振り分ける。

## 4. 受付チェックリスト（検証の中身）

依頼を受けたら、Managerは次を確認する。

### 共通チェック
- [ ] どのスタイル/ジャンルか判別できるか（できなければ質問 or 拒否）
- [ ] 担当agentが存在するか（無ければ「未対応」と返す）
- [ ] 出力枚数が現実的か（数百枚超は時間警告）
- [ ] 破壊的操作・環境変更を含まないか（含むなら人間確認）

### 立ち絵・差分agent向けチェック（最初のagent）
- [ ] キャラ定義があるか（`character_variations.json` の characters に存在 or 新規定義が来ているか）
- [ ] 参照画像（i2iの場合）のパスが指定されているか
- [ ] 表情・ポーズの指定が定義済みのキーか（未定義キーはスキップ警告）
- [ ] モード（greenback / cg）の指定があるか（無ければ greenback 既定）
- [ ] ComfyUIが起動しているか（一括生成の直前に確認）

## 5. 標準の処理フロー（対話依頼A系の例）

```
1. 受付   「サクラの表情差分つくって」
2. 解釈   ジャンル=立ち絵差分 → 担当=tachie_diff agent
3. 確認   モードは？(greenback既定) / 枚数は？(優先25組 or 全件) / 参照画像は？
4. 検証   character_variations.json に heroineA(サクラ) あり。参照画像パスOK。
5. 提示   「greenbackで優先25枚、参照=ref_sakura.png。ドライランで確認しますか？」
6. 実行   ComfyUI起動確認 → generate_variants.py をドライラン → 本実行
7. 報告   「成功 24/25。失敗1件(timeout: shy_praying)。出力: output/galge/greenback/」
```

> **必ずドライラン（件数・ファイル名確認）を挟む**。いきなり本実行しない（量産事故防止）。

## 6. 機械依頼B系の処理フロー（概要・詳細はフェーズ4）

```
1. 受付   JSONファイルを受け取る
2. 検証   JSON Schema で形式検証（必須項目・型・許可値）
   - 不合格 → 受理せず、どこが違うかを返す
3. 安全   禁止内容・上限枚数・許可パスのチェック
4. 振り分け 同上
5. 実行   ドライラン結果を依頼元に返す → 承認後に本実行（運用方針は要相談）
6. 報告   機械可読の結果JSON（成功/失敗/出力パス）を返す
```

## 7. 失敗時の扱い

| 状況 | Managerの対応 |
| ---- | ---- |
| ComfyUI未起動（接続エラー） | 即停止。「BATで起動してください」と案内 |
| 参照画像なし | 停止。正しいパスを要求 |
| 未定義の表情/ポーズキー | その項目だけスキップし、警告として報告 |
| 生成タイムアウト | その1件を失敗として記録し継続。最後にまとめて報告 |
| transformers等のノードエラー | 停止。環境レポート§4の対処を案内 |
| 危険/不適切な依頼 | 受理しない。理由を返す |
